# G8 — CIPHER UNIVERSAL TRIGGER

CIPHER activates on ALL irreversible outputs — not email only. Any output that cannot be recalled once delivered triggers the full CIPHER verification sequence.

**What this means:** G1–G7 were built from the ██████████ email failure. The pattern they define — verify account, lock recipients, one version, approval, purpose check, load-bearing details, readback — is not email-specific. It is irreversibility-specific. G8 generalizes the trigger condition.

**CIPHER now activates when any of the following are the output:**
- Email (original G1–G7 scope — unchanged)
- Financial commitments or decisions (G9)
- Code pushed to production (G10)
- Published artifacts — documents, reports, public records (G11)
- Any other output the creator explicitly designates as irreversible

**What does NOT trigger CIPHER:**
- Draft outputs held for review
- Internal notes and session logs
- Reversible file edits
- Outputs explicitly designated as staging/provisional

**Conflict note:** The CIPHER jurisdiction conflict in Document 2 is unresolved — whether CIPHER becomes a sub-protocol under RELAY or remains independent is an open V2 decision gate. G8 expands CIPHER's trigger scope while respecting its current independent status. G8 does not resolve the jurisdiction conflict — it does not place CIPHER under RELAY.

**Why this rule exists:** The pattern G1–G7 defines is the correct pattern for any irreversible output. Limiting it to email means financial decisions, production deployments, and published documents operate without the same verification discipline. That is a structural gap, not a design choice.
