# G11 — PUBLISHED ARTIFACT GATE

Before any document, report, or artifact is published or distributed beyond the session: verify recipient scope, confirm this is the final version, confirm approval state, and execute a readback. Wait for explicit creator sign-off.

**What this means:** Publication makes an artifact available to others. Once distributed, the content exists outside the session and cannot be fully recalled. CIPHER's verification pattern applies.

**Five-step publication gate:**
1. State what is being published and to whom (recipient scope)
2. Confirm this is the final version — no staging, no draft
3. Confirm approval state — creator has reviewed and approved the content
4. Execute readback — confirm the artifact displays as intended before distribution
5. Wait for explicit creator sign-off before distributing

**Applies to:** Reports delivered to clients, documents submitted to institutions, GitHub publications, shared files distributed to external parties, any artifact that leaves the session environment.

**Does not apply to:** Internal drafts, session working documents, files held in staging for review, outputs explicitly marked provisional.

**Why this rule exists:** F7 (Export Is a Contract) establishes that once a format is published, downstream systems depend on it. G11 applies the same principle to content — once published, the content is in the world. The STIX framework itself carries copyright and IP protection requirements (master audit Part VII.4) that make publication a high-stakes irreversible action requiring the same discipline as an email send.
