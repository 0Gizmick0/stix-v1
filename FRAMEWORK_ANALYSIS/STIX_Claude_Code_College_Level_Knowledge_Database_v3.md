STIX framing. This version corrects the earlier shallow treatment. It is organized as a working database: what to
study, what each source is for, what concept it unlocks, and how that concept plugs into a governed AI
operating stack. The emphasis is on using Claude and Claude Code effectively right now while building toward
systems engineering, AI engineering, retrieval-backed knowledge, and governance-aware control.

1. Executive thesis
Claude Code should not be treated as a magical coding surface that improves only by prompting harder. The real leverage comes
from separating roles and reducing waste. Browser Claude should own framing, decomposition, critique, synthesis, and final review.
Claude Code should own repository action, file edits, shell interactions, bounded automation, and implementation loops. The bridge
between them is not guesswork. It is persistent project memory, skills, hooks, MCP connectivity, and retrieval-backed access to your
own data. That is the central systems idea this database is built around.

A college-level understanding matters because the concepts you are circling are not only prompt engineering concepts. They are
systems engineering concepts, software engineering concepts, AI engineering concepts, retrieval engineering concepts, and
governance concepts. If you ignore that broader structure, Claude Code usage becomes brittle. If you include it, Claude becomes
one part of an explicit operating architecture.

2. STIX operating model for Claude use
- Use browser Claude for decision support, planning, comparison, postmortems, and synthesis.

- Use Claude Code for acting inside repos, terminals, tests, and controlled file changes.

- Use CLAUDE.md, auto memory, rules, and skills to avoid re-explaining the same project every session.

- Use hooks to preprocess machine output and logs before they reach the context window.

- Use MCP when Claude needs tool access, database access, or workflow integrations.

- Use embeddings and retrieval when Claude must work from your own documents or records reliably.

- Use software and systems engineering discipline so the stack remains maintainable and auditable.

3. Concepts missing from the earlier PDF that needed to be added
- Systems engineering was underrepresented. You need requirements discipline, interface design, validation strategy, lifecycle
thinking, and operational boundaries, not just prompt tactics.

- Software engineering was underrepresented. A serious Claude Code workflow depends on module boundaries, maintainability,
code review habits, CI/CD awareness, and failure containment.

- Governance and ethics were underrepresented. If Claude is given tools, data, and automation pathways, ethics and risk are not
optional side topics.

- RAG and embeddings were underrepresented as the correct practical path for 'training it with a database.' The system gets
stronger through externalized knowledge and retrieval, not by pretending browser chat is enough.

- Evaluation was underrepresented. Retrieval, agent behavior, and workflow value all need measurement, not just intuition.

4. College-level learning order
- First: Anthropic core usage - Claude 101, Claude Code in Action, memory, common workflows, skills, hooks, MCP.

- Second: software and systems engineering - MIT systems engineering and software engineering foundations.

- Third: governance and ethics - MIT AI ethics and responsible AI materials.

- Fourth: retrieval and AI/ML integration - embeddings, RAG, vector databases, evaluation.

- Fifth: advanced orchestration - Agent SDK, subagents, GitHub Actions, and stateful agent systems.

5. Operating conclusion
The right interpretation is this: the real project is not 'learn Claude Code.' The real project is to build a governed AI operating stack.
Browser Claude is one interface. Claude Code is one execution runtime. The knowledge system that makes them useful consists of
memory, rules, skills, hooks, MCP, retrieval, software engineering discipline, systems engineering discipline, ethics, and
measurement. That is why this version had to become broader and more academic. Without those surrounding concepts, the earlier
PDF would remain too narrow and too operationally shallow.
